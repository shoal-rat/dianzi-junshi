# 本地决策流水线

## 设计目标

这条流水线优先保证三件事：资料默认留在本机；模型不能跳过决策直接写答案；高级能力失败时仍有可用结果。

## 运行阶段

| 阶段 | 输入 | 输出 | 失败时 |
|---|---|---|---|
| 收集 | 本轮文字、图片路径、历史 | 事件与证据引用 | 保留原始输入 |
| 观察 | 当前文本与结果 | 有来源的结构化观察 | 本地规则提取 |
| 信念 | 历史观察 | 双时间尺度状态 | 使用无信息先验 |
| 假设 | 状态与冲突 | 归一化竞争解释 | 保留“未知” |
| 检索 | 目标、状态、历史 | BM25 ⊕ 向量 ⊕ RRF 融合后的多样化证据包 | 仅使用近期记录 |
| 策略 | 目标、缺失信息、EVOI | 多个策略候选 | 使用安全策略库 |
| 模拟 | 策略、假设、学习参数 | 信念空间 rollout 分支（回应分布、预测状态、值方差） | 纯先验世界模型 |
| 批评 | 分支与证据 | 多维评分 | 本地评分器 |
| 规划 | 奖励与不确定性 | 单一策略或追问 | 收住或补信息 |
| 表达 | 已选策略 | 自然语言变体 | 离线模板 |
| 学习 | 真实结果 | 世界模型、后验与状态更新 | 保留事件待重放 |

## 规划预算

- `快速`：最多 8 条证据、4 个策略、2 个体制分支、rollout 深度 1。
- `平衡`：最多 12 条证据、6 个策略、3 个体制分支、rollout 深度 2。默认设置。
- `深入`：最多 18 条证据、8 个策略、4 个体制分支、rollout 深度 3。

预算限制候选数、体制分支数和 rollout 深度，防止推理树无限展开。根节点对四类回应精确枚举，深层用确定性等价延拓收敛计算量。当前实现通常只需要一个表达模型调用；状态更新、检索、模拟与评分都在本地完成。

## 上下文管理

“无损压缩”不是可靠承诺。这里采用分层保留：

1. 原始消息和截图始终留在本地。
2. 截图逐张转成可回指记忆卡，不把整批图片塞进上下文。
3. 近期消息固定保留，较早资料按问题检索。
4. 时间事实保留有效区间；冲突事实并存，不强制合并。
5. 只有本轮最有决策价值的证据进入表达提示。

这种做法与近年的长任务上下文管理方向一致：保留外部持久记忆，主动缩短工作上下文，并让检索只带回当前有用的部分。

## 供应商适配

引擎显式检测看图、流式、本地执行、结构化输出和原生约束解码能力。支持约束解码的供应商（Anthropic 强制 tool schema、OpenAI 兼容 response_format）由解码器而不是提示词保证 JSON 合法，提示修复循环只作为回退；Codex 与 Claude Code 复用电脑现有登录；API 供应商使用自己的密钥；演示模式不联网。缺失能力不会让整条流水线失败。

## 可观测性

每次决策保存各阶段耗时、证据扫描/采用数量、模拟分支数、规划模式、模型调用数和缓存命中。右侧“完整思考过程”显示适合用户阅读的摘要，诊断接口提供机器可读导出：

```text
GET  /api/partners/:slug/decisions/latest
GET  /api/partners/:slug/decisions
GET  /api/partners/:slug/decisions/:id
GET  /api/partners/:slug/decision-diagnostics
POST /api/partners/:slug/decision-engine/rebuild
```
