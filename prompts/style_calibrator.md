# 用户风格校准器

## 任务

分析用户自己的说话习惯，确保回复听起来像**用户**说的话。

好的建议要经过两重过滤：
1. 这条回复适合**对象**吗？（来自 partner_builder）
2. 这条回复听起来像**用户**吗？（来自本文件）

---

## 采集方式

### 方式 A：用户口述（Q4 in intake.md）

用户在 Q4 中描述自己的回复风格，从描述中提取：

| 描述 | 推断行为规则 |
|------|------------|
| "话少" | 回复通常 1–3 句，不主动展开新话题 |
| "喜欢撩" | 允许加入轻微暧昧的表达 |
| "比较正经" | 不用撒娇/过于可爱的表达 |
| "常用hh" | 生成时可加入"hh"代替"哈哈哈" |
| "喜欢发表情包" | 可在回复后单独给表情包类型建议 |
| "不用表情包" | 不主动推荐表情包 |

### 方式 B：从聊天记录推断

如果有聊天记录，提取用户侧的消息分析：

- **消息密度**：用户平均发几条/次？
- **消息长度**：多少字算典型长度？
- **语气词**：用户常用什么？（好的/嗯/哦/hh/噢噢）
- **标点**：加不加句号？感叹号频率？
- **称呼**：叫对方什么？
- **表情包习惯**：用什么类型的表情包？频率？
- **幽默风格**：有没有特定的梗或说法？

---

## 用户风格档案格式

```markdown
## 用户风格档案

### 消息格式
- 消息密度：{多条连发/一次一条/视情况}
- 典型长度：{短/中/长}
- 典型例子："{用户的真实回复1}" / "{用户的真实回复2}"

### 语言习惯
- 常用语气词：{列表}
- 标点风格：{描述}
- 常用表情包/微信内置表情：{列表}
- 表情包风格偏好：{抽象/可爱-线条小狗/可爱-Chiikawa/帅气/长辈图反差/沙雕文字/不用表情包}
- 称呼对象的方式：{如：宝/你/名字}

### 语气倾向
- 总体风格：{甜蜜型/调侃型/理性型/话少型/温柔型}
- 撩的程度：{喜欢撩/偶尔/不太}
- 幽默方式：{自嘲/反差梗/无厘头/基本不开玩笑}
- 展示自己方式：{生活状态/审美品味/工作学习/运动社交/不擅长展示}
```

## 应用规则

生成回复时，应用用户风格：

1. **语气词替换**：用用户惯用的语气词替换通用语气词
   - 如用户惯用"嗯嗯"不用"好的"
   - 如用户常说"hh"不说"哈哈哈哈哈"

2. **长度控制**：如果用户话少，建议也短；用户话多，建议可以更丰富

3. **表情包一致性**：不要在用户从不用表情包的情况下主动推荐表情包；可复制回复默认只写文字

4. **撩的程度**：如果用户风格比较正经，不给过于暧昧的方案

5. **幽默对齐**：和用户的幽默风格一致，避免生成用户不会说的梗；不确定的梗先查再用

6. **展示自己对齐**：如果需要让用户展示自己，把内容包装成用户能自然说出口的方式
   - 话少用户：一句具体状态 + 一个轻问题
   - 幽默用户：轻微自嘲或反差
   - 正经用户：真实分享，不硬撩
   - 不会展示的用户：给可直接发的短句，不让他临场发挥大段

7. **机器腔过滤**：删掉用户平时不会说的整齐表达、解释腔和万能总结

## 气质频道（痞一点 / 乖一点）

用户风格是底盘，气质频道是在这个底盘上，把对方更吃的那一面**多放出来一点**。有人吃"痞、桀骜、有点坏、不好拿捏"，有人吃"乖、稳、温柔、让人安心"。同一句话换个气质说，效果差一倍。涉及气质选择时读取 `references/persona_modes.md`。

关键原则：**放大用户真实有的一面，演出来的人设撑不久。**

- 先从本文件拿到用户真实风格底盘，再在 ta **做得到、不别扭**的范围内调。完全不会怼人的用户，不硬塞痞话术，给"少回一句、留个白"的轻量版即可。
- 调的是表达方式和节奏。
- 对方气质偏好从证据判断（见 `persona_modes.md`），不足就默认走稳一点的乖向。
- 痞不等于可以更油，乖不等于可以舔；油腻度始终服从当前阶段上限（见 `prompts/oil_control.md`）。
- 翻车提醒：别从"痞"滑到爹味/忽冷忽热折磨人，别从"乖"滑到舔狗/没火花。

## 默认风格（用户没有提供时）

当没有足够信息推断用户风格时，使用以下通用默认值：
- 消息密度：中等（2–3 条）
- 长度：中等（10–25 字/条）
- 语气：自然、有温度，不过度卖萌
- 表情包：默认不配；确实需要时只给类型建议
- 撩的程度：轻微，不过分
