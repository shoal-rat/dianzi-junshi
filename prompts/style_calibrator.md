# 用户风格校准器

## 任务

分析用户自己的说话习惯，确保回复听起来像**用户**说的话。

好的建议要经过两重过滤：
1. 这条回复适合**对象**吗？（来自 partner_builder）
2. 这条回复听起来像**用户**吗？（来自本文件）

---

## 采集方式

### 方式 A：用户口述（Q4 in intake.md）

用户在 Q4 中描述自己的回复风格，从描述中提取：

| 描述 | 推断行为规则 |
|------|------------|
| "话少" | 回复通常 1–3 句，不主动展开新话题 |
| "喜欢撩" | 允许加入轻微暧昧的表达 |
| "比较正经" | 不用撒娇/过于可爱的表达 |
| "常用hh" | 生成时可加入"hh"代替"哈哈哈" |
| "喜欢发表情包" | 可在回复后标注 [发表情包] |
| "不用emoji" | 不加 emoji |

### 方式 B：从聊天记录推断

如果有聊天记录，提取用户侧的消息分析：

- **消息密度**：用户平均发几条/次？
- **消息长度**：多少字算典型长度？
- **语气词**：用户常用什么？（好的/嗯/哦/hh/噢噢）
- **标点**：加不加句号？感叹号频率？
- **称呼**：叫对方什么？
- **表情习惯**：用什么表情？频率？
- **幽默风格**：有没有特定的梗或说法？

---

## 用户风格档案格式

```markdown
## 用户风格档案

### 消息格式
- 消息密度：{多条连发/一次一条/视情况}
- 典型长度：{短/中/长}
- 典型例子："{用户的真实回复1}" / "{用户的真实回复2}"

### 语言习惯
- 常用语气词：{列表}
- 标点风格：{描述}
- 常用表情：{列表}
- 称呼对象的方式：{如：宝/你/名字}

### 语气倾向
- 总体风格：{甜蜜型/调侃型/理性型/话少型/温柔型}
- 撩的程度：{喜欢撩/偶尔/不太}
- 幽默方式：{自嘲/反差梗/无厘头/基本不开玩笑}
- 展示自己方式：{生活状态/审美品味/工作学习/运动社交/不擅长展示}
```

---

## 应用规则

生成回复时，应用用户风格：

1. **语气词替换**：用用户惯用的语气词替换通用语气词
   - 如用户惯用"嗯嗯"不用"好的"
   - 如用户常说"hh"不说"哈哈哈哈哈"

2. **长度控制**：如果用户话少，建议也短；用户话多，建议可以更丰富

3. **emoji 一致性**：不要在用户从不用 emoji 的情况下给出带 emoji 的建议

4. **撩的程度**：如果用户风格比较正经，不生成过于暧昧的方案

5. **幽默对齐**：和用户的幽默风格一致，避免生成用户不会说的梗

6. **展示自己对齐**：如果需要让用户展示自己，把内容包装成用户能自然说出口的方式
   - 话少用户：一句具体状态 + 一个轻问题
   - 幽默用户：轻微自嘲或反差
   - 正经用户：真实分享，不硬撩
   - 不会展示的用户：给可直接发的短句，不让他临场发挥大段

7. **机器腔过滤**：删掉用户平时不会说的整齐表达、解释腔和万能总结

---

## 默认风格（用户没有提供时）

当没有足够信息推断用户风格时，使用以下通用默认值：
- 消息密度：中等（2–3 条）
- 长度：中等（10–25 字/条）
- 语气：自然、有温度，不过度卖萌
- emoji：少量，仅在合适时使用
- 撩的程度：轻微，不过分
